\begin{abstract}
Roofline-based analytical models are increasingly used to guide GPU
architecture design for Large Language Model (LLM) inference. But how
reliable is this guidance? Through a systematic design space exploration
of 5,082 GPU architecture variants using the LLMCompass framework---the
most comprehensive GPU-level DSE for LLM inference to date---we provide
the first critical analysis of what roofline-based DSE actually reveals
and where it fundamentally misleads.

Our study yields four novel findings that survive a rigorous novelty
analysis against 95 recent papers across 22 research directions.
\emph{First}, we derive a precise decode latency formula
$T = 65.16/\text{BW} + 25.20$\,ms ($R^2 > 0.9999$), exposing a 25\,ms
compute floor that limits HBM bandwidth scaling returns---a result not
captured by any prior analytical model. \emph{Second}, we construct the
first complete phase-orthogonal sensitivity matrix, quantifying that
systolic array scaling provides 30.5$\times$ better area-ROI than core
scaling for prefill, while decode is entirely insensitive to compute
resources. \emph{Third}, we demonstrate that commonly-used averaging-based
sensitivity analysis produces misleading results---SRAM appears to have
8.1\% prefill impact under naive averaging but has exactly \emph{zero}
effect under controlled sweeps. \emph{Fourth}, we show that the roofline
model has a critical blind spot: on-chip memory (SRAM, global buffer)
shows zero latency sensitivity across all 5,082 designs, meaning
roofline-based DSE \emph{cannot guide memory hierarchy design}---a
fundamental limitation for the community to address.

We distill these findings into six design guidelines for next-generation
LLM accelerators and identify the boundary between what analytical DSE
can and cannot reliably inform.
\end{abstract}

\keywords{GPU Architecture, LLM Inference, Design Space Exploration,

\section{Introduction}
\label{sec:intro}

Large Language Model (LLM) inference has become one of the most important
workloads in modern datacenters. The scale of deployment---GPT-4 reportedly
serves hundreds of millions of queries per day---makes architectural
efficiency directly impact cost and carbon
footprint~\cite{stojkovic2024greener,llmco2}. As GPU vendors design
next-generation accelerators with LLM inference as a first-class design
target, the community needs rigorous methodology to navigate the vast
design space.

Analytical models, particularly roofline-based
frameworks~\cite{williams2009roofline,verhelst2025rooflines}, are
attractive tools for design space exploration (DSE) because they enable
rapid evaluation of thousands of design points without expensive cycle-level
simulation. Recent work has produced several such
frameworks~\cite{llmcompass,pope2023efficiently,zigzag2021}, and they
have been used to inform architectural decisions including
phase-specialized chip design~\cite{spad2025}, chip architectures under
export control constraints~\cite{ning2025sanctions}, and disaggregated
serving systems~\cite{splitwise2024,distserve2024}.

However, a critical question remains largely unexamined: \emph{how
reliable is the design guidance produced by roofline-based DSE?} The
roofline model is an intentional abstraction that trades accuracy for
speed. Understanding where this abstraction faithfully represents
architectural tradeoffs---and where it fundamentally misleads---is
essential for the community.

In this paper, we present the most comprehensive GPU-level DSE study for
LLM inference to date: 5,082 architecture variants evaluated across
61 workload configurations spanning 5 model sizes. Our analysis yields
\emph{both} positive results (reliable design guidance in several
dimensions) \emph{and} negative results (critical blind spots that the
community must address). We make the following contributions:

\begin{enumerate}
\item \textbf{Decode latency formula with compute floor.}
We derive $T = 65.16/\text{BW} + 25.20$\,ms ($R^2 > 0.9999$), revealing
that 56\% of the fastest decode latency is a compute floor insensitive to
bandwidth scaling. No prior work has identified this floor or its
implications for HBM investment ROI (Section~\ref{sec:decode_formula}).

\item \textbf{Phase-orthogonal sensitivity matrix with area-ROI.}
We construct the first complete 5-parameter $\times$ 3-objective
sensitivity matrix using controlled single-parameter sweeps, showing
systolic array scaling provides 30.5$\times$ better area-ROI than core
scaling for prefill. This quantitative map enables principled
phase-specialized design (Section~\ref{sec:sensitivity}).

\item \textbf{Methodology critique: averaging produces artifacts.}
We demonstrate that the common practice of averaging across configurations
at each parameter level~\cite{llmcompass} produces misleading sensitivity
estimates. SRAM appeared to have 8.1\% prefill impact under naive
averaging but has exactly zero effect under controlled sweeps
(Section~\ref{sec:methodology_critique}).

\item \textbf{Roofline model blind spot for memory hierarchy.}
We show that on-chip memory parameters (SRAM 64--512\,KB, global buffer
20--120\,MB) have \emph{zero} effect on latency across all 5,082 designs.
This reveals a fundamental limitation: roofline-based DSE cannot guide
on-chip memory sizing decisions, despite memory hierarchy being critical
for techniques like FlashAttention~\cite{dao2022flashattention}
(Section~\ref{sec:blind_spot}).
\end{enumerate}

Contributions 1--2 provide actionable design guidance; contributions 3--4
identify boundaries of the methodology itself. Together, they establish
what GPU architecture DSE can and cannot reliably inform for LLM inference.

\section{Background}
\label{sec:background}

\subsection{LLM Inference Phases}

Transformer-based LLM inference proceeds in two phases with fundamentally
different computational profiles~\cite{splitwise2024,pope2023efficiently}.
The \textbf{prefill} phase processes the input prompt through all
transformer layers, performing large matrix multiplications on tensors of
shape $(b, s, d)$ that are typically compute-bound. The \textbf{decode}
phase generates tokens autoregressively, processing tensors of shape
$(b, 1, d)$ per step, which is memory-bandwidth-bound due to the
weight-dominated memory access pattern.

This duality is well-established in the
literature~\cite{splitwise2024,distserve2024,spad2025} and has motivated
phase-disaggregated serving~\cite{splitwise2024,distserve2024,mooncake2024}
and phase-specialized architectures~\cite{spad2025}. Our work does not
re-establish this duality but rather asks: \emph{given this duality, what
quantitative design guidance can analytical DSE provide, and where does
it fall short?}

\subsection{Roofline Performance Model}

The roofline model~\cite{williams2009roofline} characterizes operator
performance by arithmetic intensity (AI = FLOPs/bytes):
\begin{equation}
T = \max\left(\frac{\text{FLOPs}}{\pi}, \frac{\text{Bytes}}{\beta}\right)
\end{equation}
where $\pi$ is peak compute throughput and $\beta$ is peak memory
bandwidth. Recent extensions target ML
accelerators~\cite{verhelst2025rooflines} and sparse
workloads~\cite{shinn2023sparsity}. The model's key strength is that it
enables rapid evaluation of design points; its key weakness is that it
uses peak bandwidth regardless of on-chip data reuse patterns.

\subsection{DSE Frameworks for LLM Inference}

LLMCompass~\cite{llmcompass} provides GPU-level co-simulation with
operator-level roofline modeling and a process-aware area cost model.
Pope et al.~\cite{pope2023efficiently} present an analytical model for
transformer inference on TPUs. ZigZag~\cite{zigzag2021} and
Timeloop~\cite{parashar2019timeloop} target spatial accelerator DSE with
dataflow mapping optimization. Ning and Wentzlaff~\cite{ning2025sanctions}
apply LLMCompass-based DSE under export control constraints.
SPAD~\cite{spad2025} uses LLMCompass to design phase-specialized prefill
and decode chips. Unlike these works, which use DSE to \emph{find optimal
designs}, we use DSE to \emph{analyze the design landscape and evaluate
the methodology itself}.

\subsection{Prefill-Decode Disaggregation}

The fundamental difference between prefill and decode phases has driven
a large body of work on disaggregated serving.
Splitwise~\cite{splitwise2024} separates prefill and decode onto different
GPU types, achieving superior cost-efficiency. DistServe~\cite{distserve2024}
disaggregates prefill and decode for goodput-optimized serving, reporting
up to 7.4$\times$ more requests. Sarathi-Serve~\cite{agrawal2024sarathi}
introduces chunked prefills to tame throughput-latency tradeoffs.
Mooncake~\cite{mooncake2024} proposes a KV-cache-centric disaggregated
architecture. SPAD~\cite{spad2025} takes disaggregation to the hardware
level, designing specialized prefill chips (larger systolic arrays + GDDR)
and decode chips (high HBM bandwidth, reduced compute).

Our work complements this line of research by providing the \emph{quantitative
sensitivity matrix} that justifies disaggregation: we show exactly which
parameters matter for each phase and by how much, enabling principled
parameter allocation for phase-specialized designs.

\subsection{LLM Serving Systems}

Orca~\cite{yu2022orca} introduced iteration-level scheduling for
continuous batching of transformer-based serving.
vLLM~\cite{kwon2023efficient} applies virtual memory management to KV
caches via PagedAttention, achieving near-zero memory waste.
Llumnix~\cite{sun2024llumnix} enables dynamic scheduling with live
migration of inference requests across GPU instances.
DeepSpeed-Inference~\cite{aminabadi2022deepspeed} provides multi-GPU
inference with heterogeneous CPU-GPU strategies. These systems implement
optimizations (batching, memory management, scheduling) that affect
utilization patterns; our DSE captures the \emph{architectural}
tradeoffs that determine the performance envelope within which these
systems operate.

\subsection{Attention and Memory Optimization}

FlashAttention~\cite{dao2022flashattention} demonstrates that IO-aware
tiling of attention computation can achieve 2--4$\times$ speedup by
exploiting on-chip SRAM---precisely the kind of optimization that our
roofline model \emph{cannot capture} (Section~\ref{sec:blind_spot}).
FlashAttention-2~\cite{dao2023flashattention2} and
FlashAttention-3~\cite{shah2024flashattention3} further improve
parallelism and low-precision support.
GQA~\cite{ainslie2023gqa} reduces KV cache size by grouping query heads,
changing the arithmetic intensity landscape.
KV cache compression~\cite{yang2024kvcompress,hooper2025squeezed,xiao2024duoattention}
reduces the memory footprint of decode, potentially shifting the
memory-compute balance our DSE analyzes.

\subsection{Quantization for Inference}

SmoothQuant~\cite{xiao2023smoothquant} enables W8A8 quantization by
migrating quantization difficulty from activations to weights.
GPTQ~\cite{frantar2023gptq} and AWQ~\cite{lin2024awq} push to 3--4 bit
weight quantization with minimal quality loss. These techniques
effectively double or quadruple the arithmetic intensity of
weight-memory-bound operators, potentially shifting decode from
memory-bound to compute-bound. Our DSE uses FP16; extending to
quantized workloads is important future work that would test whether
the decode latency formula still holds.

\subsection{Alternative Hardware Approaches}

Processing-in-Memory (PIM) addresses the decode memory bottleneck at
the hardware level. NeuPIMs~\cite{heo2024neupims} proposes NPU-PIM
heterogeneous acceleration. CXL-based approaches~\cite{gu2025pim,park2024cxlpnm,kim2025lia}
extend the memory hierarchy beyond HBM. Chiplet
architectures~\cite{shao2021simba,chipletgym2024} enable separate
compute and memory chiplets. These approaches address the \emph{same}
decode bottleneck our analysis quantifies but through hardware
innovation rather than architectural parameter tuning.

\section{Methodology}
\label{sec:methodology}

\subsection{Simulation Framework}

We use LLMCompass~\cite{llmcompass} to model GPU systems at the operator
level. Each system comprises one or more devices connected by an
interconnect. Each device contains a compute module (with configurable
core count and systolic arrays), an IO module (with configurable L2 cache
and HBM channels), and memory. For each operator, the framework computes
$T = \max(\text{FLOPs}/\pi, \text{bytes}/\beta)$ using a process-aware
area cost model (7nm TSMC).

\subsection{Workload Suite}

We profile 5 transformer models spanning 3 orders of magnitude in
parameter count (Table~\ref{tab:models}): GPT-2 Small (117M), GPT-2 XL
(1.5B), LLaMA-7B, LLaMA-70B, and GPT-3 175B. Each model is evaluated
across 10 workload scenarios varying batch size (1--64), sequence length
(128--8192), phase (prefill/decode), and device count (1/4/8), yielding
\textbf{61 profiling tasks with 732 operator-level measurements}.

\subsection{Sensitivity Analysis Methodology}
\label{sec:sensitivity_method}

We evaluate sensitivity using \emph{controlled single-parameter sweeps}:
varying one parameter while holding all others at baseline values
(64 cores, SA~$16{\times}16$, HBM~4, 128~KB SRAM, 60~MB global buffer).
We explicitly compare this against the naive averaging method (computing
mean latency at each parameter level across all configurations) to
demonstrate the methodological difference (Section~\ref{sec:methodology_critique}).

\section{Results}
\label{sec:results}

\subsection{Bottleneck Taxonomy}
\label{sec:bottleneck}

Across 732 operator measurements, we confirm the phase duality reported
in prior work~\cite{splitwise2024,distserve2024}: prefill operators are
a mix of compute-bound (32.8\%) and memory-bound (67.2\%), while decode
operators are \textbf{universally memory-bound} (100\%, all 384
measurements). This holds across all 5 models, all batch sizes, all
sequence lengths, and all device counts.

The key addition over prior assertions is \emph{operator-level granularity}:
we identify that bottleneck classification is determined by \emph{operator
type}, not workload parameters. FFN matmuls and QKV projections are always
compute-bound in prefill (AI $>$ 1500); attention operators and vector
operations are always memory-bound in both phases.

The sensitivity matrix reveals \textbf{near-perfect orthogonality}
between prefill and decode optimization vectors:
\begin{itemize}
\item Prefill is dominated by systolic array dimension (1253\% range),
with secondary contribution from core count (277\%). HBM channels have
only 2.9\% effect.
\item Decode is dominated solely by HBM channels (73.5\%). All other
parameters---including systolic array dimension and core count---have
\emph{exactly zero} effect.
\item The cross-influence (SA on decode, HBM on prefill) is negligibly
small (0.0\% and 2.9\% respectively).
\end{itemize}

\paragraph{Area-ROI comparison.}
The area efficiency of systolic array scaling versus core scaling is
dramatically different (Figure~\ref{fig:area_efficiency}):

\begin{itemize}
\item SA dim 8$\to$32: 13.53$\times$ prefill speedup for 12.9\% area
increase $\Rightarrow$ \textbf{ROI = 104.9$\times$/area-unit}
\item Cores 32$\to$128: 3.77$\times$ prefill speedup for 109.8\% area
increase $\Rightarrow$ \textbf{ROI = 3.4$\times$/area-unit}
\end{itemize}

The ratio is \textbf{30.5$\times$} in favor of systolic array scaling.
This occurs because systolic throughput scales quadratically with
dimension ($O(n^2)$) while die area scales linearly, creating a
super-linear ROI that core count replication cannot match.

No prior work has quantified this area-ROI comparison for LLM inference
architectures. SPAD~\cite{spad2025} designs phase-specialized chips but
does not report per-mm$^2$ efficiency. Chip Under
Sanctions~\cite{ning2025sanctions} explores the design space under
constraints but does not isolate individual parameter contributions
with controlled sweeps.

\begin{figure}[t]
\centering
\includegraphics[width=\linewidth]{figures/area_efficiency.pdf}
\caption{Area-ROI comparison: systolic array dimension scaling versus
core count scaling for prefill latency. SA provides 30.5$\times$ better
return per mm$^2$ of die area.}
\label{fig:area_efficiency}
\end{figure}

\subsection{Decode Latency Formula and Compute Floor}
\label{sec:decode_formula}

We discover that decode latency follows a precise hyperbolic model
(Figure~\ref{fig:decode_formula}):
\begin{equation}
T_{\text{decode}} = \frac{65.16}{\text{BW}} + 25.20 \text{ ms}
\quad (R^2 > 0.9999)
\label{eq:decode_formula}
\end{equation}
where BW is HBM bandwidth in TB/s.

The 25.20\,ms constant represents a \textbf{compute floor}---irreducible
latency from non-memory operations (compute pipeline, scheduling,
synchronization). At the highest tested bandwidth (3.28~TB/s, HBM=8),
this floor constitutes \textbf{55.9\%} of total decode time.

\paragraph{Implications for bandwidth investment.}
The hyperbolic form means that doubling bandwidth from 1.6~TB/s to
3.3~TB/s reduces decode latency by only 29\% (from 65.9~ms to 45.1~ms),
not 50\%. Further doubling to 6.6~TB/s would yield only 12\% additional
reduction (to 35.1~ms). The floor imposes severe diminishing returns on
HBM investment.

This formula has not appeared in prior work. Pope et
al.~\cite{pope2023efficiently} model transformer inference analytically
on TPUs but do not derive a closed-form decode latency with a compute
floor. Ma and Patterson~\cite{ma2026memory} identify memory bandwidth
as the primary challenge but do not quantify the floor. Our formula
provides a concrete tool for architects to evaluate the ROI of bandwidth
investment.

The 25.20\,ms compute floor (dashed line) constitutes 56\% of latency

\subsection{Pareto Frontier Analysis}

Of 5,082 designs, 79 are Pareto-optimal across prefill latency, decode
latency, and die area. The frontier structure confirms the phase
orthogonality (Figure~\ref{fig:pareto}):
\begin{itemize}
\item \textbf{4-device designs} (48/79) achieve the best decode (34.5~ms)
but higher prefill ($\geq$766.7~ms).
\item \textbf{8-device designs} (31/79) achieve the best prefill
(550.5~ms) but worse decode ($\geq$47.7~ms).
\item No 1-device design appears on the Pareto frontier.
\item Prefill-decode correlation on the frontier is $r = -0.294$
(anti-correlated).
\end{itemize}

This anti-correlation provides quantitative support for the disaggregated
serving approach of Splitwise~\cite{splitwise2024} and
DistServe~\cite{distserve2024}.

\centering
\includegraphics[width=\linewidth]{figures/pareto_frontier.pdf}
\caption{Pareto frontier across 5,082 designs. No 1-device design is
Pareto-optimal. 4-device designs favor decode; 8-device designs favor

\label{fig:pareto}
\end{figure}

\subsection{Multi-Device Scaling}

Decode scaling is negative beyond 4 devices: going from 4 to 8 devices
\emph{increases} decode latency by 25\% (0.51~ms $\to$ 0.64~ms for
LLaMA-70B) because allreduce communication overhead exceeds the compute
savings. Prefill scaling remains reasonable (78.2\% efficiency at 8
devices). This asymmetry further supports phase-specific device count
selection.

\subsection{DSE Methodology Critique}
\label{sec:methodology_critique}

We compare two sensitivity analysis approaches on the same 5,082-design
dataset (Table~\ref{tab:method_compare}):

\begin{table}[t]
\centering
\caption{Naive averaging vs. controlled sweeps for sensitivity analysis.
Naive averaging produces phantom sensitivities for on-chip memory.}
\label{tab:method_compare}
\begin{tabular}{lrr}
\toprule
\textbf{Parameter} & \textbf{Naive (\%)} & \textbf{Controlled (\%)} \\
\midrule
Systolic array & 1,074.8 & 1,253.3 \\
Core count     & 206.2   & 277.2 \\
HBM channels   & 35.2    & 73.5 \\
Global buffer  & 17.4    & \textbf{0.0} \\
SRAM           & 8.1     & \textbf{0.0} \\
\bottomrule
\end{tabular}
\end{table}

The naive method produces \emph{phantom sensitivities}: SRAM appears to
have 8.1\% impact and global buffer 17.4\% impact under averaging, but
both have \emph{exactly zero} effect under controlled sweeps. The phantom
effect arises because designs with larger SRAM/buffer also tend to have
different core counts and SA dimensions, and the averaging method
attributes the correlated latency variation to the wrong parameter.

This finding has direct implications for prior DSE work that uses
averaging-based sensitivity analysis~\cite{llmcompass}: reported
sensitivities for confounded parameters may be entirely artifactual.
We recommend that all DSE studies use controlled single-parameter sweeps
with explicit baseline configurations.

\subsection{Roofline Blind Spot: On-Chip Memory}
\label{sec:blind_spot}

The zero sensitivity of SRAM and global buffer is not merely a statistical
artifact---it is a \emph{structural property of the roofline model}.
Because the model computes operator latency as
$\max(\text{FLOPs}/\pi, \text{bytes}/\beta)$ using \emph{peak} HBM
bandwidth $\beta$, it assumes all data is served from HBM regardless of
on-chip availability. Varying SRAM from 64~KB to 512~KB (8$\times$) or
global buffer from 20~MB to 120~MB (6$\times$) produces identical latency
predictions.

In reality, on-chip memory is critical for performance:
\begin{itemize}
\item FlashAttention~\cite{dao2022flashattention} achieves 2--4$\times$
speedup by tiling attention computation to fit in SRAM.
\item Global buffer (L2 cache) provides data reuse for weight
matrices~\cite{jia2019dissecting}.
\item KV cache residency in on-chip memory eliminates repeated HBM
accesses during decode~\cite{kwon2023efficient}.
\end{itemize}

This blind spot means that roofline-based DSE \emph{should not be used
to dimension on-chip memories}. For memory hierarchy design, the community
needs models that capture data reuse, multi-level caching, and
tiling~\cite{parashar2019timeloop,zigzag2021}---or must resort to
cycle-accurate simulation.

\section{Design Guidelines}
\label{sec:guidelines}

We distill our findings into six guidelines, each annotated with its
reliability given the roofline model's limitations:

\begin{enumerate}
\item \textbf{Prioritize systolic array dimensions over core count}
(\emph{high confidence}). SA scaling provides 30.5$\times$ better
area-ROI than core scaling. This is reliable because the roofline
model accurately captures compute throughput scaling.

\item \textbf{Target 3--4~TB/s HBM bandwidth; beyond this, diminishing
returns dominate} (\emph{high confidence}). The decode formula
(Eq.~\ref{eq:decode_formula}) quantifies the floor.

\item \textbf{Use 64--96 core count as the balanced mid-range}
(\emph{moderate confidence}). Beyond 96 cores, area cost severely
outweighs latency benefit. Area model is reliable; latency model may
underestimate benefits from improved scheduling.

\item \textbf{Optimize for 4-device, not 8-device decode serving}
(\emph{moderate confidence}). Communication model captures the
inversion point but may not reflect NVLink 5/NVSwitch advances.

\item \textbf{Do not rely on roofline DSE for on-chip memory sizing}
(\emph{critical finding}). SRAM and buffer show zero modeled effect
but are known to matter in practice. Use cycle-accurate simulation.

\item \textbf{Consider prefill-decode architectural disaggregation}
(\emph{high confidence}). The near-perfect orthogonality of
sensitivity vectors is a robust finding.
\end{enumerate}

%% ===========================================================================
\section{Discussion}
\label{sec:discussion}

\paragraph{Scope and limitations.}
Our study uses FP16 arithmetic. Quantization
(W8A8~\cite{xiao2023smoothquant}, W4A16~\cite{lin2024awq,frantar2023gptq})
would change arithmetic intensity and may shift the decode phase toward
compute-bound, potentially invalidating the decode formula. We use
multi-head attention (MHA); GQA~\cite{ainslie2023gqa} reduces KV cache
size and would affect decode memory access patterns. MoE
models~\cite{hwang2024pregated,tutel2023} have sparse activation
patterns creating different hardware demands. Very long sequences
($>$8K tokens) with KV-cache management may change the on-chip memory
calculus. Speculative decoding~\cite{leviathan2023fast} transforms
decode from single-token to small-batch verification.

\paragraph{Generalizability of the decode formula.}
The formula $T = 65.16/\text{BW} + 25.20$\,ms is derived from a specific
workload (LLaMA-70B, bs=16, sl=1024) on the LLMCompass model. The
\emph{functional form} (hyperbolic with floor) should generalize because
the floor reflects non-memory compute overhead, but the \emph{coefficients}
will vary with model size, batch size, and sequence length. We encourage
the community to validate this formula on real silicon.

\paragraph{Complementarity with prior work.}
Our sensitivity matrix complements SPAD's~\cite{spad2025}
phase-specialized designs by providing the quantitative justification.
Our methodology critique applies to any roofline-based DSE, including
ZigZag~\cite{zigzag2021} and Timeloop~\cite{parashar2019timeloop},
when used without on-chip memory modeling.

%% ===========================================================================
\section{Conclusion}
\label{sec:conclusion}

We present the first critical analysis of roofline-based GPU architecture
DSE for LLM inference. Through 5,082 design evaluations, we derive a
precise decode latency formula with a compute floor, construct a
phase-orthogonal sensitivity matrix showing 30.5$\times$ better area-ROI
for systolic array scaling, demonstrate that common sensitivity analysis
methodology produces phantom sensitivities, and identify a critical blind
spot in on-chip memory modeling. These findings establish the boundary
between what analytical DSE can reliably inform (compute and bandwidth
allocation) and what requires more detailed modeling (memory hierarchy
design). We hope this work helps the community use roofline-based DSE
more effectively and motivates the development of more accurate analytical
models that capture on-chip data reuse.
